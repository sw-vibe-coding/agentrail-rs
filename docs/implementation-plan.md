# Implementation Plan

This document replaces the Phase 0-4 plan from `continue-from-bootstrap.md`
with a revised plan reflecting the two-layer architecture.

## Completed

- 8 CLI commands: init, status, next, begin, complete, plan, history, abort
- File-based persistence (TOML + JSON + JSONL)
- Saga lifecycle: create, advance, complete
- Step role typing (Meta, Production, Deterministic, Validation, Legacy)

## Phase 1: ICRL Core Loop

**Goal**: make `agentrail next` inject past successful trajectories into its
output. This is the single highest-ROI change -- the ICRL payoff with
minimal refactoring.

### 1a. Add task_type to steps

Add an optional `task_type: Option<String>` field to `StepConfig`. This is
the key that links a step to its skill and experience libraries.

- Update `StepConfig` in `agentrail-core`
- Add `--task-type <type>` flag to `complete` command (for setting task_type
  on the next step)
- Update `create_step()` to accept task_type
- Add tests

### 1b. Wire trajectory retrieval into `next`

When `agentrail next` encounters a step with a non-null `task_type`:

1. Call `retrieve_successes()` for that task type
2. Format results as structured examples in the output
3. Include between the prompt section and the "when done" section

Format:

```
=== PAST SUCCESSES (tts) ===
[run_005] action=gradio_client, result=ok, reward=+1
[run_003] action=gradio_client, result=ok, reward=+1
```

- Update `next` command
- Add tests with pre-populated trajectory data

### 1c. Record experiences on complete

When `agentrail complete` finishes a step that has a `task_type`:

1. Prompt or accept experience metadata (actions taken, result, reward)
2. Save as a trajectory record
3. Use existing `save_trajectory()` infrastructure

- Add `--reward <-1|0|1>` and `--actions <description>` flags to `complete`
- Update tests

**Estimated scope**: ~200 lines of code changes, ~10 new tests.

## Phase 2: Dual Memory (Skills + Experiences)

**Goal**: implement the XSkill dual-memory pattern with structured skill
documents and enriched experience records.

### 2a. Skill type and storage

- Add `Skill` type to `agentrail-core` (procedure, success_patterns,
  common_failures, output_contract)
- Add `skills/` storage in `agentrail-store` (load/save TOML per task_type)
- Skills can be seeded manually or generated by distill

### 2b. Experience type (enriched trajectory)

- Add `Experience` type extending `Trajectory` with trigger conditions,
  structured actions list, failure_mode, notes
- Add `experiences/` storage alongside existing `trajectories/`
- Backward-compatible: old trajectories readable as minimal experiences

### 2c. Skill injection into `next`

- When a skill doc exists for the step's task_type, inject the full
  procedure, success patterns, and failure modes into `next` output
- Skill content appears before experience examples

### 2d. Distill command

- `agentrail distill <task_type>`: analyze experiences, generate/update
  skill document
- Groups experiences by success/failure
- Extracts common action patterns and failure modes
- Writes versioned skill TOML

### 2e. Update tests

- Skill round-trip tests
- Experience round-trip tests
- Distill logic tests
- Integration test: complete with experience -> distill -> next shows skill

**Estimated scope**: ~500 lines new code, ~15 new tests.

## Phase 3: Domain Repo Support

**Goal**: enable per-domain knowledge repos with executors, validators,
and curated skill/experience libraries.

### 3a. Domain registry

- Add `domains.toml` config to `.agentrail/`
- Domain discovery: read domain.toml manifests from registered paths
- Task-type-to-domain mapping

### 3b. Executor trait and shell executor

- Define `Executor` trait in `agentrail-core`
- Implement shell executor: pipes JobSpec params as JSON to stdin,
  reads result JSON from stdout
- Route `agentrail run-next` to the correct domain executor by kind

### 3c. Validator trait and shell validator

- Define `Validator` trait in `agentrail-core`
- Implement shell validator: same stdin/stdout JSON interface
- Wire into `complete` or a new `agentrail validate` command

### 3d. Merged retrieval

- Skill retrieval merges local + domain repo skills (local overrides)
- Experience retrieval merges local + domain repo experiences
- Domain repo experiences are read-only from Layer 1's perspective

### 3e. Contribute command

- `agentrail contribute <task_type> --to <domain>`: copy local experiences
  to domain repo for curation

### 3f. First domain repo

- Create `agentrail-domain-media` as a separate GitHub repo
- Seed with skills for TTS, ffmpeg-concat, video-composite, probe
- Include shell executors and validators
- Prove the full loop: domain skill + local experience -> next -> agent
  succeeds -> complete with reward -> distill updates skill

**Estimated scope**: ~800 lines new code in agentrail-rs, new repo for
domain-media, ~20 new tests.

## Phase 4: Knowledge Graph Validation

**Goal**: optional structured reward signals from task-type tool chains.

### 4a. Graph definition format

- TOML schema for directed graphs (edges with from/to/required)
- Load from domain repo `graphs/` directory

### 4b. Graph-based reward computation

- Compare experience action sequence against expected graph
- Compute structured reward: which required edges were followed/skipped
- Replace bare i8 reward with `RewardDetail` type

### 4c. Graph-aware distillation

- Distill command uses graph structure to identify systematic deviations
- Failure modes auto-categorized by which graph edge was violated

**Estimated scope**: ~400 lines, ~10 tests.

## Phase 5: Hybrid Orchestrator Loop

**Goal**: auto-advance through deterministic steps, escalate to agent only
on failure or for semantic work.

- Meta -> Production -> Deterministic -> Validation cycle
- `agentrail run-loop`: execute all deterministic steps in sequence,
  pause at production/meta steps for agent
- Resume after interruption
- Record experience on every step completion

**Estimated scope**: ~300 lines, ~10 tests.
