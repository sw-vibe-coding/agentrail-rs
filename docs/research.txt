2 — Tiny Rust ICRL Demo
This is something you could actually show on screen.

The goal is to simulate trajectory learning.

Data Model
use serde::{Serialize, Deserialize};

#[derive(Serialize, Deserialize, Clone)]
struct Trajectory {
    state: String,
    action: String,
    reward: i32,
}
Fake agent memory
fn best_action(state: &str, history: &[Trajectory]) -> Option<String> {
    history
        .iter()
        .filter(|t| t.state == state)
        .max_by_key(|t| t.reward)
        .map(|t| t.action.clone())
}
Example usage
fn main() {
    let history = vec![
        Trajectory {
            state: "tts".into(),
            action: "curl wrong endpoint".into(),
            reward: -1,
        },
        Trajectory {
            state: "tts".into(),
            action: "tts_cli --server localhost:8888".into(),
            reward: 1,
        },
    ];

    let action = best_action("tts", &history);

    println!("Agent chooses: {:?}", action);
}
Output:

Agent chooses: "tts_cli --server localhost:8888"

trajectory history → policy decision

8. TDD methodology: practical loop

A. Write the contract first

#[test]
fn generate_tts_job_builds_expected_command_line() { ... }
B. Make the test tiny and specific
One behavior per test.

Do not test:

parsing

storage

validation

execution

rendering

all in one monster. That way lives madness.

C. Prefer pure logic first
Push side effects to edges.

Good candidates for pure functions:

command construction

validation rules

packet rendering

retrieval scoring

Pure functions are cheap to test and hard to lie about.

D. Wrap all side effects
Create traits/interfaces for:

command runner

clock

ID generator

agent invocation

artifact probing

Then test with fakes.

That gives you reliable tests instead of “works on my machine under a waxing moon.”
